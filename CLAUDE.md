# CLAUDE.md (프로젝트 전용)

이 문서는 `~/.claude/CLAUDE.md`(전역 규칙) 전체를 그대로 가져오지 않는다. 이 프로젝트는
**Bowling Game Kata**를 TDD로 연습하기 위한 것으로, 전역 규칙 중 아래 항목만 차용한다.

- TDD 개발 방법론 (Red → Green → Refactor)
- 작업 단위별로 plan/result 문서를 작성하는 관행 — 단, 저장 위치는 전역 규칙의
  `docs/tasks/`가 아니라 **`docs/` 바로 아래**로 변경한다 (예: `docs/2026-07-14_xxx_plan.md`).
- 코드 주석은 영어로 작성한다.

그 외 전역 CLAUDE.md의 세부 규칙(CONCEPTS/DETAILS/TROUBLESHOOTING/LESSONS/CHANGELOG 문서
세트, 아키텍처 원칙 등)은 이 프로젝트에 강제하지 않는다.

참고: `../harness` 프로젝트는 doc-consistency-verifier/ai-action/test-verifier/
compliance-verifier 4개 서브에이전트로 구성된 "Verify Harness" 파이프라인까지 갖춘 더 무거운
버전이다. 이 프로젝트는 그보다 단순한 "Original TDD + AI 보조" 버전이며, 사람이 Red-Green-
Refactor 사이클을 직접 몰고 AI를 구현 보조 도구로 사용한다.

## 1) 프로젝트 개요

이 프로젝트의 목적은 Robert C. Martin이 고안한 **Bowling Game Kata**를 TDD로 연습하는
것이다. 미국식 텐핀 볼링(American Ten-Pin Bowling) 한 게임의 유효한 투구 순서가 주어졌을
때, 해당 게임의 최종 총점을 계산하는 로직을 구현한다. 실제 볼링 게임(공 굴리기, 핀
시뮬레이션, UI)을 만드는 것이 아니라, **투구 결과(쓰러뜨린 핀 수)를 입력받아 최종 점수를
계산하는 핵심 로직**에만 집중한다.

구현 대상은 `Game` 클래스 하나이며, 다음 2개의 메서드만 가진다 (원본 kata 시그니처는
`void roll(int pins)` / `int score()`이며, 이 프로젝트에서는 Python 관례에 맞게 아래처럼
옮긴다).

- `roll(pins)`: 플레이어가 공을 던질 때마다 호출된다. `pins`는 쓰러뜨린 핀의 수다.
- `score()`: 게임의 총점을 계산하여 반환한다.

### 범위 밖 (Out of Scope)

가벼운 연습(kata)에 집중하기 위해 아래 사항은 고려하지 않는다. 실제 애플리케이션이라면
필요하겠지만, 여기서는 핵심 로직에 집중하기 위해 의도적으로 생략한다.

- 투구가 유효한지 검증 (예: `pins`가 0~10 범위인지 확인하는 로직)
- 투구 횟수나 프레임 수가 정확한지 검증
- 중간 프레임의 점수 제공 (프레임별 중간 점수 조회 API)
- 실제 게임 진행(공 굴리기 애니메이션, 핀 배치 시뮬레이션), CLI/UI

## 2) Bowling 점수 규칙 (Single Source of Truth)

구현 중 규칙에 대한 의문이 생기면 아래 내용을 기준으로 판단한다. 이 절이 이 프로젝트의
명세서 역할을 한다.

볼링 게임은 총 10개 프레임으로 구성된다. 각 프레임에서 플레이어는 10개의 핀을 쓰러뜨리기
위해 두 번의 투구(Roll) 기회를 가지며(매 프레임 시작 시 핀은 10개로 초기화된다), 10번째
프레임은 예외적으로 최대 3번까지 투구할 수 있다. 프레임 점수는 쓰러뜨린 총 핀 수에
스트라이크와 스페어에 따른 보너스를 더해 계산한다.

- **기본 점수**: 한 프레임에서 얻는 점수는 해당 프레임에서 쓰러뜨린 핀의 총합이다.
- **스페어(Spare)**: 두 번의 투구로 10개의 핀을 모두 쓰러뜨린 경우다. 해당 프레임의
  보너스는 **다음번 투구(1회)**에서 쓰러뜨린 핀의 수다.
- **스트라이크(Strike)**: 첫 번째 투구에서 10개의 핀을 모두 쓰러뜨린 경우다. 이 경우
  해당 프레임은 한 번의 투구로 종료된다. 해당 프레임의 보너스는 **이후 두 번의 투구**에서
  쓰러뜨린 핀 수의 합계다.
- **10번 프레임**: 마지막 10번 프레임에서 스페어나 스트라이크를 기록하면 프레임을
  완수하기 위한 추가 투구를 할 수 있다 (스페어 → 1번 추가, 스트라이크 → 2번 추가, 총 최대
  3투구). 이 추가 투구 자체가 다시 스트라이크/스페어이더라도, 그로 인한 별도의 추가 투구
  기회는 없다.

### 예시

- `[3, 2]` (일반 프레임) → 해당 프레임 점수 = 3 + 2 = 5
- `[2, 8]` (스페어, 다음 투구가 3) → 해당 프레임 점수 = 2 + 8 + 3 = 13
- `[10]` (스트라이크, 다음 두 투구가 3, 5) → 해당 프레임 점수 = 10 + 3 + 5 = 18
- 퍼펙트 게임 (12번 모두 스트라이크, 마지막 프레임 보너스 2회 포함) → 총점 300

## 3) 구현 방식

- 산출물은 순수 로직 모듈 하나(`Game` 클래스)로 충분하다. `print`/`input`을 호출하지
  않는다. CLI나 별도 어댑터 계층을 임의로 추가하지 않는다 (요청받지 않은 레이어링 금지).
- 파일 구조:
  ```
  game.py              # Game 클래스 (roll, score) — 순수 로직
  tests/
  └── test_game.py
  ```
- TDD 순서: 요구사항(기본 점수 → 스페어 → 스트라이크 → 10번째 프레임 → 퍼펙트 게임) 하나당
  **실패하는 테스트를 먼저 작성**하고, 그 테스트를 통과시키는 **최소한의 구현**을 추가한 뒤,
  동작을 바꾸지 않는 범위에서 **리팩터링**한다.
- 다음 테스트가 요구하기 전까지 코드를 미리 추가하지 않는다. 한 번에 여러 규칙을 앞질러
  구현하지 않는다.

## 4) TDD 커밋 컨벤션

이 프로젝트는 TDD(Red → Green → Refactor)로 진행하며, 각 단계마다 커밋 제목 앞에 아래
태그를 붙인다.

|단계|태그|의미|
|---|---|---|
|Red|`test:`|실패하는 테스트를 먼저 작성|
|Green|`feat:`|테스트를 통과시키는 최소 구현|
|Refactor|`refactor:`|동작 변경 없이 구조 개선|

테스트 파일명 규칙: 대상 모듈이 프로젝트 루트의 `game.py`이므로, 테스트 파일은
`tests/test_game.py`로 대응한다.

## 5) plan/result 문서 규칙

- 코드 변경을 동반하는 작업은 구현 착수 전에 `docs/yyyy-mm-dd_<summary>_plan.md`를
  작성하고, 테스트 통과를 확인한 뒤 `docs/yyyy-mm-dd_<summary>_result.md`를 작성한다.
  (전역 규칙의 `docs/tasks/` 하위가 아니라 `docs/` 바로 아래에 둔다.)
- 코드 변경이 없는 작업(예: 이 문서 자체의 수정, 규칙 논의)은 plan/result 문서를 만들지
  않는다.
- plan/result 문서 본문은 한국어로 작성하고, 기술 용어는 원어를 유지해도 된다.
