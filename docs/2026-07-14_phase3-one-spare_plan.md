# Phase 3 — 스페어 하나 구현 계획

Written at (KST): 2026-07-14 15:25

## 1. 작업 목적

지금까지의 구현(`score = sum(rolls)`)은 보너스가 없는 게임만 올바르다. 스페어가 하나
있을 때 보너스(다음 1투구)가 정확히 더해지는지 확인하고, 그 로직을 구현한다.

## 2. 현재 상태 (독자 관점)

- `Game.score()`는 모든 투구의 핀 수를 단순히 합산한다.
- 스페어가 있으면 다음 투구 핀 수를 보너스로 더해야 하는데, 지금은 더해지지 않는다
  (수동 확인: 5,5,3,나머지0 → 현재 결과 13, CLAUDE.md 규칙상 기대값 16).

## 3. 목표 상태

- 첫 프레임이 스페어(5,5)이고 다음 투구가 3일 때, 총점이 16
  (프레임1: 5+5+3보너스=13, 프레임2: 3+0=3, 나머지 0)이 되도록 한다.

## 4. 대표 시나리오

- 투구 순서: `5, 5, 3, 0×17` → `score() == 16`

## 5. 제안하는 변경 접근

- `score()`를 프레임 단위로 순회하도록 바꾼다. 각 프레임에서 두 투구의 합이 10이면
  스페어로 판단해 다음 1투구를 보너스로 더한다. 아직 스트라이크는 이 phase의 범위가
  아니다(다음 phase에서 다룬다).

## 6. 예상 설명 구조

결과는 `docs/2026-07-14_phase3-one-spare_result.md`에 정리한다.

## 7. 모호성 / 가정 / 트레이드오프 / 리스크

- **가정**: 이 phase에서는 스트라이크를 별도로 분기하지 않는다. 스트라이크가 섞인
  입력은 다음 phase(스트라이크)에서 다룬다.
- **리스크**: "두 투구 합이 10"이라는 판정만으로는 스트라이크(첫 투구가 10)가 뒤섞인
  경우 오판할 수 있으나, 이번 phase의 테스트는 스트라이크를 포함하지 않으므로 영향이
  없다.

## 8. 검증 방법

- `PYTHONPATH=. pytest -q`로 새 테스트가 실패(Red) → 통과(Green)하는지 확인하고, 기존
  테스트(거터 게임, 모두 1핀)가 계속 통과하는지 함께 확인한다.
