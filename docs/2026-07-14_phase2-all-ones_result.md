# Phase 2 — 모두 1핀(보너스 없는 일반 프레임) 구현 결과

Written at (KST): 2026-07-14 15:16

## 1. 한눈에 보는 요약

- "모두 1핀 → 총점 20" 케이스를 테스트로 명세해 두었다.
- 계획 문서(plan)에서 예상한 대로, 이 phase는 **구현 변경 없이 테스트 추가만으로 즉시
  통과(Green)했다.** 보너스(스페어/스트라이크)가 없는 게임에서는 총점이 정의상 "쓰러뜨린
  핀 수의 총합"과 같으므로, Phase 1의 구현이 이미 이 케이스를 만족시킨다.

## 2. 왜 필요했는가

보너스 로직을 추가하기 전에, "보너스가 없는 일반적인 프레임 조합"에서도 점수 계산이
맞는다는 것을 테스트로 남겨 회귀(regression) 안전망을 만들기 위함이다.

## 3. Before / After

- **Before**: "거터 게임"(전부 0핀) 하나만 명세되어 있었다.
- **After**: "거터 게임"과 "모두 1핀" 두 가지 무보너스 시나리오가 함께 명세되어 있다.
  구현(`game.py`)은 변경되지 않았다.

## 4. 대표 시나리오

- 20번의 투구가 모두 1핀 → `score()` 결과는 `20`.
  (`tests/test_game.py::test_all_ones_scores_twenty`)

## 5. 동작/구조 변화

- 없음. `game.py`는 Phase 1과 동일하다.

## 6. 영향 범위

- 없음(구현 변경이 없으므로 기존 동작에 영향 없음). 다음 phase(스페어)에서 실제로
  `score()`의 계산 방식이 확장될 예정이다.

## 7. 제약 / 후속 작업 / 알려진 한계

- 이 phase는 CLAUDE.md의 "Red → Green 쌍 = 1 phase" 원칙을 문자 그대로는 만족하지
  못한다 — 테스트가 실패한 적이 없기 때문이다. 대신 "테스트만 있고 구현 변경이 없는
  phase"로 투명하게 기록한다.
- 다음 phase(스페어)부터는 현재의 "핀 총합 반환" 구현으로는 실패하는 실제 Red가
  발생함을 미리 확인했다 (수동 확인: 5,5,3,0×17 투구 → 실제 13, 기대 16).

## 8. Lessons

- **교훈**: Bowling Kata에서 "보너스 없는 일반 프레임" 테스트는, 이전 phase의 최소 구현
  (핀 총합 반환)이 이미 수학적으로 옳기 때문에 새 코드를 요구하지 않는 경우가 흔하다.
  이런 경우 억지로 실패를 유도하거나 구현을 불필요하게 바꾸지 말고, "구현 변경 없이 통과"
  했다는 사실을 결과 문서에 그대로 기록하는 것이 정직한 TDD 기록 방법이다.
