# Phase 5 — 10번째 프레임 보너스 구현 결과

Written at (KST): 2026-07-14 15:42

## 1. 한눈에 보는 요약

- 10번째 프레임에서 스트라이크/스페어가 나왔을 때의 보너스 투구 처리를 테스트로
  명세해 두었다.
- Phase 2와 같은 패턴으로, **구현 변경 없이 테스트 추가만으로 즉시 통과(Green)했다.**
  `score()`의 프레임 순회가 정확히 10회로 고정되어 있어, 보너스 투구는 항상 다음 프레임이
  아니라 lookahead(다음 1~2투구)로만 읽히기 때문이다.

## 2. 왜 필요했는가

10번째 프레임은 볼링 규칙에서 예외적으로 추가 투구를 허용하는 유일한 프레임이다. 이
동작이 우연이 아니라 알고리즘 설계상 보장된다는 것을 테스트로 남겨 회귀를 방지한다.

## 3. Before / After

- **Before**: 10번째 프레임의 보너스 투구 처리가 테스트로 명세되어 있지 않았다.
- **After**: 10번째 프레임 스트라이크(+보너스 2투구)와 스페어(+보너스 1투구) 두 시나리오가
  명세되어 있다. `game.py`는 변경되지 않았다.

## 4. 대표 시나리오

- 1~9프레임 모두 거터, 10번째 프레임 스트라이크 + 보너스 5, 3 → `score() == 18`.
  (`tests/test_game.py::test_tenth_frame_strike_gets_two_bonus_rolls`)
- 1~9프레임 모두 거터, 10번째 프레임 스페어(5,5) + 보너스 4 → `score() == 14`.
  (`tests/test_game.py::test_tenth_frame_spare_gets_one_bonus_roll`)

## 5. 동작/구조 변화

- 없음. `game.py`는 Phase 4와 동일하다.

## 6. 영향 범위

- 없음(구현 변경이 없으므로 기존 동작에 영향 없음). 다음 phase(퍼펙트 게임)도 같은
  이유로 구현 변경 없이 통과할 것으로 예상된다.

## 7. 제약 / 후속 작업 / 알려진 한계

- 이 phase도 Phase 2와 마찬가지로 "Red → Green 쌍" 원칙을 문자 그대로 만족하지 못했다
  (테스트가 실패한 적이 없음). 결과 문서에 그 사실을 투명하게 기록했다.

## 8. Lessons

- **교훈**: "정확히 10번 순회하며 lookahead로 보너스를 읽는" 설계는 스트라이크/스페어
  분기(Phase 3, 4)를 구현하는 순간 10번째 프레임의 특수 규칙까지 부수적으로 만족시킨다.
  이는 이 kata의 잘 알려진 특징이며, 별도의 "10번째 프레임 전용 분기"를 추가하지 않아도
  된다는 뜻이다. 미리 특수 분기를 만들지 않고 기존 lookahead 로직을 신뢰한 것이 맞는
  선택이었다.
