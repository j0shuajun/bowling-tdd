# Phase 5 — 10번째 프레임 보너스 구현 계획

Written at (KST): 2026-07-14 15:40

## 1. 작업 목적

10번째 프레임에서 스페어/스트라이크가 나오면 추가 투구(보너스 투구)가 주어진다.
이 보너스 투구가 정확히 반영되고, 동시에 그 보너스 투구 자체가 다시 스트라이크/
스페어이더라도 별도의 추가 투구 기회가 생기지 않는지 확인한다.

## 2. 현재 상태 (독자 관점)

- `Game.score()`는 정확히 10번(프레임 수만큼) 순회하며, 각 프레임에서 스트라이크/
  스페어 여부에 따라 다음 1~2투구를 lookahead로 더한다.
- 수동 확인 결과, 10번째 프레임이 스트라이크(+보너스 5,3)인 경우와 스페어(+보너스 4)인
  경우 모두 이미 기대값과 일치한다 (18, 14).

## 3. 목표 상태

- 10번째 프레임의 스트라이크/스페어 보너스 케이스를 테스트로 명세해 회귀를 방지한다.

## 4. 대표 시나리오

- 프레임1~9 모두 거터, 10번째 프레임 스트라이크 + 보너스 5, 3 → `score() == 18`.
- 프레임1~9 모두 거터, 10번째 프레임 스페어(5,5) + 보너스 4 → `score() == 14`.

## 5. 제안하는 변경 접근

- 테스트만 추가한다. `score()`의 순회가 정확히 10회로 고정되어 있어, 보너스 투구는
  항상 lookahead로만 읽히고 새로운 프레임으로 취급되지 않는다. 즉 Phase 4의 구현이
  이미 이 규칙을 만족시킬 것으로 예상된다.

## 7. 모호성 / 가정 / 트레이드오프 / 리스크

- **예상되는 예외 상황**: Phase 2와 마찬가지로, 이 phase도 구현 변경 없이 즉시 통과할
  가능성이 높다. 순회 횟수가 고정된 lookahead 알고리즘의 자연스러운 결과이기 때문이다.
- 실제 실행 결과에 따라 결과 문서의 서술을 확정한다.
